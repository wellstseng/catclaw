# Skill：<skill-name>（Inversion）

> Agent 主導對話，先把所有必要需求問清楚才開始執行。**未問完禁止動作。**

## 觸發

- 使用者要求 `<任務類型>` 但條件不足
- 任何「資訊缺一項就會做錯」的場景

## 核心鐵則

1. **未走完訪談禁止進入執行階段** — 不准「邊做邊問」
2. **一次一題，等回答 → 等確認 → 才下一題**
3. **不腦補預設值** — 答不出來就停下，請使用者決定
4. 訪談分階段：每階段完才能進下一階段

## 工作流程

### Phase 1：問題探索（基本需求）
依序問，前題未答完不問後題：
- Q1：要解決什麼具體問題？
- Q2：使用者 / 受眾是誰？
- Q3：成功的衡量標準？

**完成判定**：3 題都有實質回答（非「不知道」「隨便」）

### Phase 2：條件限制（技術 / 資源）
- Q4：執行環境 / 平台限制？
- Q5：技術堆疊 / 工具要求？
- Q6：絕對不能違反的條件？

**完成判定**：3 題都有實質回答

### Phase 3：綜合與確認
- 用收集到的需求填 `assets/spec-template.md`
- 給使用者看完整需求單 → 詢問是否準確
- 使用者確認後才進入下一個 skill（通常接 Generator / Pipeline）

## 必備檔案

| 檔 | 用途 |
|----|------|
| references/question-bank.md | 各階段問題集（可依任務類型分組） |
| assets/spec-template.md | 訪談結果填寫的需求單 |

## 反模式

- ❌ 一次問 5 題（使用者懶得逐個答 → 隨便回 → 失敗）
- ❌ 「我假設你的意思是 X」這種腦補
- ❌ 缺答案還是硬進執行階段
- ❌ 收集完不給 review，直接動手

## 注意事項

- 推薦組合：**Inversion → Generator**（先問完再生成）
- 訪談 token 成本高於執行 token — 但比「執行錯誤重做」便宜
